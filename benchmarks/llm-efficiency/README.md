# LLM Efficiency Benchmark — scrml vs. React+TS

**Purpose:** Measure whether scrml is structurally more efficient for LLM-assisted development than incumbent stacks. The hypothesis is that scrml's compile-time wiring (auto-await, reactive deps, engine transitions, exhaustive match, render-spec dispatch) reduces the surface where an LLM can hallucinate or omit boilerplate, producing more working code per token and fewer iteration cycles.

The benchmark either validates that hypothesis (giving scrml adopter-pitch ammunition) or invalidates it (giving scrml a course-correction signal). **Both outcomes are useful**; this isn't a marketing exercise.

---

## The hypothesis being tested

> Given the same task, the same model, and equivalent baseline knowledge, an LLM produces working-on-first-try code more often in scrml than in React+TypeScript, and uses fewer tokens to do it.

**What "equivalent baseline knowledge" means:** the React+TS side gets a minimal system prompt (React 18 + TypeScript 5 + Tailwind, that's it — these are training-set defaults the model already knows). The scrml side gets the full kickstarter v2 + primer + anti-patterns brief in its system prompt because the model has zero training data for scrml. The asymmetric prompt size is **itself a measured axis** — scrml needs more prompt today because no training corpus exists; the benchmark surfaces that cost honestly rather than hiding it.

If scrml-with-large-prompt still beats React+TS-with-tiny-prompt on tokens-per-working-feature and correctness, the structural-efficiency claim holds. If it doesn't, the claim needs revision or the language needs more compile-time guarantees.

## Measurement axes

### Quantitative

| Axis | Definition | Unit |
|---|---|---|
| **Input tokens** | System prompt + user prompt + (for retries) conversation history | tokens |
| **Output tokens** | Total tokens generated by the model across all iterations | tokens |
| **Wall time** | First-message-sent to acceptance-decision | seconds |
| **Iterations to working** | How many human-equivalent "fix this" turns before the code passes validation | integer (1 = first try) |
| **Compile success rate** | Of N samples, what fraction produces code that compiles | 0.0-1.0 |
| **Runtime success rate** | Of compiled samples, what fraction passes the spec's runtime assertions | 0.0-1.0 |
| **Bundle size** | Size of compiled output that ships to the browser | bytes |
| **Source LOC** | Substantive lines (blank + comment stripped) | integer |
| **Type errors at compile** | Errors emitted by the type system | count |

### Qualitative (recorded but not auto-scored)

- **Idiomatic patterns used** — did the model reach for the canonical shape (e.g., engine over booleans-as-lifecycle in scrml; `useReducer` over `useState` chains in React) — recorded by manual review
- **Hallucinated APIs** — did the model invent functions/methods/keywords that don't exist — recorded by validator failure-mode classification
- **Workaround invention** — when the model couldn't express something canonically, did it invent a workaround vs. fail loudly — recorded by manual review

---

## What the benchmark is NOT

- **Not a "best model wins" comparison.** Different models will rank differently per axis; the benchmark is per-axis per-language, not a single overall score.
- **Not a "best language wins" verdict.** scrml losing on some axes (e.g., wall time, due to large system prompt parsing) is expected and useful signal. React+TS losing on others (e.g., type errors, due to mutable defaults) is also expected.
- **Not bias-free.** The author of this benchmark wrote scrml. The benchmark design itself is biased toward scrml's structural-claim surfaces. **This is acknowledged up front.** External replication with adversarial spec design is welcome and the benchmark harness is designed to make that possible.

---

## What the benchmark IS

- A reproducible script. Anyone with API keys can re-run it.
- An honest measurement. The asymmetric-prompt-size cost is reported; the bias is acknowledged; the spec is adversarial-replaceable.
- A baseline that re-runs over time. Each scrml release + each model release re-runs the benchmark; trends become visible.
- A public artifact. The benchmark spec, prompts, and results are all open. The LLM-generated outputs are saved alongside the results so reviewers can inspect them directly.

Frontier tier (where the strongest claims are made):

- Claude Haiku 4.5

Three samples per (model × spec × language) trial for variance signal. Total trials per spec: 7 models × 2 languages × 3 samples = **42 trials per spec**.

## Specs (initial set)

1. **Triage board** (`specs/01-triage-board.md`) — drag-and-drop 3-column kanban. Exercises: state machine (drag phase), iteration with event capture, reactive lists, basic CSS.
2. *(planned)* Multi-step wizard with validation — exercises engine + validators + auto-synth validity surface
3. *(planned)* Real-time chat with channels — exercises channels (scrml) vs WebSocket plumbing (React)
4. *(planned)* Debounced search with cancellation — exercises debounced primitive (scrml) vs manual hooks (React)
5. *(planned)* Login + auth phase — exercises auth recipe + engine

Three to five specs gives meaningful cross-cut. Single-spec results are anecdote; cross-spec results are signal.

## Open design questions (deferred to first-run analysis)

- **Retry budget per trial:** how many "fix this error" iterations before declaring a trial failed? Default proposed: 3. Higher value rewards models that can self-correct; lower value rewards models that get it right first.
- **Whether to count system-prompt tokens as input tokens.** Argument for: real adopter cost includes prompt. Argument against: caches amortize this. Plan: report both — with-prompt and without-prompt token counts.
- **Whether to share retry context between languages.** I.e., when an LLM fails on scrml and gets an "error: X" message, does it see the same error message format the React+TS retries see? Plan: yes — both retries pass back the compiler/runtime error output verbatim.
- **Whether scrml's large system prompt should be cached.** Anthropic's prompt caching exists; using it amortizes the per-call cost for scrml's ~50k system prompt. Plan: use caching where available, report cache-hit/miss rates as a separate axis.

## What this benchmark does NOT prove (even if scrml wins)

- That scrml is mature enough for production use today
- That adopters will switch to scrml
- That the benchmark accurately reflects the adopter experience (it measures one LLM-mediated dimension; adopters care about many)
- That scrml's structural claims hold for app shapes outside the spec set

Even a clean win on this benchmark is **necessary-but-not-sufficient** evidence. The benchmark removes one specific source of doubt; it doesn't replace shipping real apps and the boring work of corpus-building.
