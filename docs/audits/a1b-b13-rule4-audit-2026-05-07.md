## §1 Findings

### §1.1 [SUBSTANTIVE — SCOPE TIMING DRIFT] §55.14 says "parse-time"; SCOPE puts B13 in A1b (resolve-type)

**SPEC §55.14 line 24691:**
> "The compiler emits `E-DERIVED-WITH-VALIDATORS` (§34) at parse-time."

**SCOPE row B13 wording:** "Step 5 left them as syntactic but A1b rejects" — i.e., B13 fires the rejection at A1b (resolve-type stage, NOT parse-time).

**Drift analysis:** the spec says parse-time; SCOPE says A1b. These are different stages. Why the gap?

The answer is operational: at parse-time (Stage 2 BS / ast-builder), the parser is intentionally permissive (Step 5 collected validators on every Shape-2-shaped decl, including derived ones). The shape-discriminant logic that tags a decl as "derived" (`shape: "derived"`, `isConst: true`) runs at parse-time but the **validator-vs-derived semantics** (you can't combine them) is a higher-level contract that A1b enforces.

**Spec follow-up flagged:** §55.14's "parse-time" wording is operationally imprecise — should say "compile-time, no later than A1b resolve-type" or specifically "A1b stage, fired during validator-walking pass." The semantic intent (it's a static error, not a runtime one) is preserved by either timing; the difference matters only to dispatch implementers.

**B13 implementation guidance:** fire `E-DERIVED-WITH-VALIDATORS` at A1b. Walk every state-decl with `_cellKind` derivable from B5's annotation; if `decl.isConst === true` AND `decl.validators?.length > 0`, fire. (Per primer §13.7 B6 specifics: `decl.isConst:true` is the canonical derived-cell discriminator.)

**Recommend:** PA flag a small spec-prose follow-up to clarify §55.14's timing wording (parallel to §6.6.10 footnote precedent). Not blocking B13.

### §1.2 [SUBSTANTIVE — SCOPE BUNDLES TWO INDEPENDENT TASKS] Two distinct work items in B13

**SCOPE B13 row** bundles:
- **(a) E-DERIVED-WITH-VALIDATORS rejection** — derived cells with validator attrs.
- **(b) 4-level error-message resolution-chain recording** — per-validator inline-override / project-registered / data-defaults / match-escape decision.

These are TWO INDEPENDENT work items. (a) is a walker that fires an error code; (b) is metadata recording on validators for A1c codegen consumption.

**Could B13 split?** Per primer §12 depth-of-survey-discount precedent: when SCOPE bundles, audit checks if the bundle is logically coherent or operationally separable.

**Analysis:**
- (a) is small (~30-60min once B5 cellKind is reliable + walker exists; B8's PASS 6 walker may be the natural home).
- (b) is more substantial — requires walking each validator and recording which level applies (inline if Level-1 override present; otherwise compute at codegen time).

**B13 brief recommendation:** keep bundled. Both fire from a single walker pass that visits every validator on every state-decl. Walker dispatches:
- If decl is derived (`isConst: true`) AND has validators → fire E-DERIVED-WITH-VALIDATORS, skip rest of validator processing on this decl.
- Else, for each validator: record inline-override if present (Level 1); record level-2-fallback marker for A1c.

Single pass; modest expansion. Cost stays in the 4-6h envelope.

### §1.3 [SCOPE-CLARIFICATION] What "recording" means in §55.10 chain

**SPEC §55.10 line 24607-24609:**
> "Resolution function — `messageFor(errorTag)`: auto-imported via `use scrml:data`, walks Levels 1 → 2 → 3 automatically. If a Level-1 inline override is present for the specific (field, validator) pair, it wins. Otherwise check the project registration."

**Level-1 inline override syntax (§55.10 line 24565):**
```scrml
<name req("Please enter your name") length(>=2, "Name must be at least 2 chars")> = <input/>
```
Per-field, per-validator. Static-string only.

**B13 implementation guidance for "recording":**
- Walk each `validator: ValidatorEntry` on each state-decl.
- If the validator has an inline string-override argument (last arg per the §55.10 convention is the override string), extract it onto the validator record as `inlineOverride: string | null`.
- Static-string-only check (per L12 Edge F — no expression interpolation): if the override is anything other than a string literal, fire `E-VALIDATOR-INLINE-DYNAMIC` (or equivalent — verify §34 catalog naming) — see §1.4 below.
- Annotation feeds A1c codegen, which emits the `messageFor` lookup logic.

### §1.5 [BOUNDARY — ENGINE STATE-CELL EXCEPTION] §55.14 says legal-but-typically-redundant for engine cells

**SPEC §55.14 line 24681-24687:**
> "Validators on engine auto-declared variables: ... universal-core vocabulary applies (e.g., `oneOf([...])` makes sense). HOWEVER, the engine's `rule=` contract is the canonical mechanism for constraining engine state... Validators on engine cells are LEGAL but typically REDUNDANT when `rule=` is the constraining contract."

**B13 implementation guidance:** do NOT fire `E-DERIVED-WITH-VALIDATORS` on engine state-cells. Engine cells are NOT derived (they're not `const <x>`). The walker check is `isConst: true` AND `validators.length > 0` — engine auto-declared cells are not `isConst`, so they pass through silently.

If B14 (engine binding) introduces a `_cellKind: "engine"` annotation later, the walker should still skip those — engine validators are legal per §55.14.

### §1.6 [BOUNDARY — DERIVED ENGINE EXCEPTION] `<engine derived=expr>` is also rejected

**SPEC §55.14 line 24689:**
> "Validators on derived cells (`const <x ...>` or `<engine derived=>`): REJECTED."

**B13 implementation guidance:** the rejection covers BOTH:
- `const <x req> = expr` — Shape-3 derived with validator attr.
- `<engine for=Phase derived=expr ...>` with validator attrs.

The first is the common case; the second is rarer. B13's walker must visit both — engine declarations are not state-decls in the same shape, so the walker may need to dispatch on engine kind separately. **Phase-0 survey item:** does B14 (engine binding) ship before or after B13? If B13 is dispatched before B14, the engine-derived case may not be reachable yet (no engine-decl annotation to dispatch on). Defer to a B13.5 or B14 follow-up if so.

### §1.7 [REUSABILITY — REFINEMENT-TYPE PRIMITIVE EQUIVALENT] §55.14 cross-ref

**SPEC §55.14 line 24692-24694:**
> "If the developer wants validation on a derived value, they should add a refinement type (`const <x>: number(>=0) = ...`) — that is the type-level invariant equivalent."

**B13 brief should include:** the error message for `E-DERIVED-WITH-VALIDATORS` SHOULD suggest the refinement-type alternative. Per spec wording: "Did you mean `const <x>: number(>=0) = ...`? Refinement-type predicates are the type-level invariant for derived values."

This is user-facing message quality, not an enforcement extension. Modest brief addition.
