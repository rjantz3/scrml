## ¬ß1 Findings

### ¬ß1.1 [SUBSTANTIVE ‚Äî REGISTRATION ARCHITECTURE] Engine cell needs a registration path through B1's symbol table

**SCOPE row** describes B14 as "engine binding" but says nothing about HOW the auto-declared cell is registered.

**SPEC ¬ß51.0.C (line 20106-20140):** the engine declaration auto-declares a reactive state cell. `<marioState>` is readable everywhere via canonical `@marioState` access, AND the engine OWNS its variable (`E-ENGINE-VAR-DUPLICATE` if separately declared).

**Architectural decision required for B14:**

- **Option A:** register the engine cell through B1's existing `StateCellRecord` registration path with a new `_cellKind` value (e.g., `"engine"`). Single registration mechanism for ALL reactive cells; downstream consumers (B3 `@name` resolution, B7 dep-tracker, etc.) automatically work.
- **Option B:** separate `EngineCellRecord` shape registered in a parallel data structure on the file-level `Scope`. More isolation; downstream consumers need engine-specific code paths.
- **Option C:** hybrid ‚Äî engine cell IS a `StateCellRecord` with `_cellKind: "engine"` plus a sibling `_engineMeta: EngineMetadata` annotation carrying engine-specific data (variants, rules, transitions).

**PA recommendation: Option C.** Reuses B1/B3/B5/B7 infrastructure (zero duplicate effort) while preserving room for engine-specific metadata. The `_cellKind` extension is small; downstream passes that need engine-specific behavior dispatch on `_cellKind === "engine"`.

**Required B14 implementation expansion:**
- New `_cellKind` value `"engine"` registered in B5's classifier.
- New `_engineMeta` annotation on the state-decl node (or a sibling EngineDeclNode if engines have a distinct AST kind).
- Engine cell is `isConst: true` semantically (the engine OWNS the variable; user code cannot reassign ‚Äî uses `.advance()` instead per ¬ß51.0.G). But B7's derived-cell dep tracker should NOT walk the engine's "RHS expression" ‚Äî engines have no `expr` to derive from (except derived engines per ¬ß51.0.J, which DO; see B16 audit).
- B3's `@marioState` resolution returns the engine cell's `StateCellRecord`; existing `_resolvedStateCell` annotation works without change.

**Cost impact:** ~1-2h added to B14 for the cellKind extension + B5 tweak. Survey-discount potential if B5's switch already handles "no shape match" ‚Üí engine fallback cleanly.

### ¬ß1.2 [SPEC-CLARIFICATION] Auto-derived var-name rule edge cases

**SPEC ¬ß51.0.C table (line 20111-20116):**

| Type name | Auto-derived var name |
|---|---|
| `MarioState` | `marioState` |
| `LoadPhase` | `loadPhase` |
| `MarioMachine` (legacy) | `marioMachine` (literal lowercase-first; `Machine` suffix kept) |
| `Health` | `health` |

**Rule:** "Lowercase-first-character of the type name."

**Edge cases NOT explicitly covered:**

- **All-uppercase types** like `URL`, `IO`, `HTTP`. Apply the rule literally ‚Üí `uRL`, `iO`, `hTTP`. Likely undesired (devs would expect `url`, `io`, `http`).
- **Single-letter types** like `T`, `Q`. ‚Üí `t`, `q`. Probably fine.
- **Types starting with digit/underscore** ‚Äî illegal per scrml's identifier rules (would fail at type-decl parse), so unreachable.
- **Multi-word lowercase types** like `myType` ‚Äî already lowercase; rule is no-op. Should fire `E-ENGINE-VAR-CONVENTION` warning suggesting PascalCase type names? Or accept silently? **Spec ambiguous.**

**B14 implementation guidance:**
- Implement the lowercase-first-character rule literally per spec.
- For all-uppercase types: spec says "lowercase-first-character" ‚Äî only the first character lowers. Per literal reading, `URL` ‚Üí `uRL`. PA Phase-0 survey: confirm this is the intended behavior or recommend a small spec amendment for "if the second character is also uppercase, lowercase a contiguous run."
- For lowercase-leading types: identity (no error). Don't fire a warning unless the user-voice signals that's desired ‚Äî spec doesn't mandate.

**Spec follow-up flagged:** ¬ß51.0.C should explicitly enumerate edge-case rules (all-uppercase, all-lowercase, single-letter, leading underscore). Footnote-level addition.

### ¬ß1.3 [WAVE-ORDERING] Cross-file mount (M18) depends on MOD's exportRegistry

**SCOPE row:** "cross-file mount via `<EngineName/>` (M18)"

**SPEC ¬ß51.0.D (line 20162-20188):** cross-file engine import uses `import { MarioMachine } from './engines.scrml'` then mount via `<MarioMachine/>` self-closing tag at the use-site. Singleton semantics across all use-sites in the importing file.

**SPEC ¬ß21.8 (M18):** `pinned` legal on imports of engines. Build on B4's `importBindings` registry ‚Äî per primer ¬ß13.7 B4 specifics line "Form 1 `export <engine var=‚Ä¶>` desugars to `export const`, indistinguishable from arbitrary const today; B14 / M18 cross-file engine import lands engine-aware export-registry annotation."

**Drift analysis:** B14 is the FIRST consumer that needs MOD's `exportRegistry` to distinguish engine exports from arbitrary `const` exports. B4 already laid the groundwork (`E-IMPORT-PINNED-INVALID` accepts `const`/`let` imports as best-effort fallback per audit Option A). B14 needs MOD to expose engine-kind annotations on exported items.

**Required B14 implementation expansion:**
- B14 reads MOD's `exportRegistry` for the imported module name (per import binding from B4).
- For each `<EngineName/>` self-closing PascalCase use-site tag in markup: lookup the import binding (B4's `lookupImportBinding`) ‚Üí if export kind is `"engine"`, treat as engine mount; if export kind is anything else, fire an error (`E-ENGINE-MOUNT-NOT-ENGINE` or similar ‚Äî Phase-0 survey verifies ¬ß34 catalog naming).
- Singleton semantics: do NOT register a new engine cell at the use-site; the mount tag references the imported singleton.

**MOD enhancement scope:** does B14's dispatch include the MOD enhancement (engine-kind in exportRegistry), or is that a separate sub-step? Phase-0 survey item.

**PA recommendation:** include the MOD enhancement in B14's brief. It's a small extension (annotate exported `<engine>` decls with kind=engine) and B14 needs it to function. Defer ONLY if survey reveals MOD requires substantial restructuring.

### ¬ß1.4 [BOUNDARY] `initial=` attribute lint is B15 territory, not B14

**SPEC ¬ß51.0.E (line 20190-...)**: `initial=.Variant` required on non-derived engines; lint `W-ENGINE-INITIAL-MISSING` if omitted (defaults to first variant).

**SCOPE row B14:** doesn't mention `initial=` ‚Äî that's B15's "engine state-child exhaustiveness + transition rule typer" or could be a separate B14.5.

**B14 implementation guidance:** B14 owns the binding + variable registration; B15 (or follow-up) owns the `initial=`/state-child/rule= machinery. B14 should record (on the engine state-decl node) the `initial=` attribute's presence + value for B15 to consume ‚Äî but NOT validate it.

**Boundary clarification for the brief:** B14 records what's there; B15 enforces.

### ¬ß1.5 [BOUNDARY] Components-vs-engines distinction (M20) is partially B14, partially B17

**SPEC ¬ß51.0.K (line 20410-20430):** components are multi-instance; engines are singleton. Component body that contains `<engine>` declaration ‚Üí `E-COMPONENT-ENGINE-SCOPE`.

**SCOPE row B17:** explicitly owns `E-COMPONENT-ENGINE-SCOPE`.

**Drift analysis:** B14 declares + binds the engine cell; if the engine declaration is INSIDE a component body, B14 must either (a) refuse the registration and let B17 fire later, OR (b) refuse outright at B14 with a deferred-to-B17 fire.

**PA recommendation:** B14 fires `E-COMPONENT-ENGINE-SCOPE` itself when it walks the engine declaration site and finds it inside a component body. The error code is owned by B17 in SCOPE wording but the FIRE SITE is wherever the violation is detected. Avoiding two passes that both walk the same nodes for the same violation reduces walker churn.

**Brief clarification:** B14 walks engine declaration sites; if surrounded by a component body scope, fires `E-COMPONENT-ENGINE-SCOPE` immediately. B17 owns the broader "components-vs-engines distinction" scope (which includes other violations B14 doesn't touch).
