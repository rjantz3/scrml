### Â§A.2 Block-splitter â€” `compiler/src/block-splitter.js`

**What's there today:**

- Tracks balanced-bracket parsing for nested structures.

**v0.next gap:**

**Cost:** ~10-15h. The `<NAME>` recognition is the load-bearing change.
**Dependencies:** lexer (Â§A.1) is a no-op precondition.
**Risk:** **HIGH.** Block-splitter is critical-path; any regression breaks the entire pipeline.

### Â§A.3 Body pre-parser â€” `compiler/src/body-pre-parser.ts` (354 LOC)

### Â§A.4 JS expression parsing (acorn) â€” `compiler/src/expression-parser.ts` (2,559 LOC) + `meta-checker.ts`

**What's there today:**
- Uses **acorn** (`import * as acorn from "acorn"`).
- Two custom acorn plugins: `scrmlAtPlugin` (handles `@var` as a single identifier token) + `scrmlEnumPlugin` (handles `.Variant` as a string token).
- Parses JS-level expressions inside scrml expressions: function calls, member access, arithmetic, ternary, JSX-shaped fragments inline, etc.
- Used in two places: `expression-parser.ts` (primary; ~all JS expression parsing) + `meta-checker.ts` (identifier extraction with regex fallback).

**v0.next gap:**
- **acorn does NOT need to be replaced.** v0.next's `<NAME> = RHS` decl form is at the SCRML statement level inside `${...}` logic blocks, not at the JS expression level. Once block-splitter + body-pre-parser extract the decl form ABOVE acorn's level, the residual JS expression (the RHS) is parseable by acorn unchanged.
- Example: `${ <count> = 0; @count + 1 }` â€” pre-processor extracts `<count> = 0` as a decl statement, leaves `@count + 1` for acorn. acorn parses `@count + 1` as a binary expression (with `@` plugin handling the `@`-prefix).
- Edge: Shape 3 `const <doubled> = @count * 2` â€” pre-processor extracts the LHS structure (`const <doubled> =`), leaves `@count * 2` for acorn. Same pattern.
- Edge: Shape 2 `<userName req length(>=2)> = <input type="text"/>` â€” pre-processor extracts decl + RHS markup; the markup itself is parsed by markup-parser (already in the pipeline), not by acorn. acorn never sees `<input/>`.
- Edge: validators with cross-cell args `<confirm req eq(@signup.password)>` â€” `eq(@signup.password)` is a function-call-shaped predicate; arg expression `@signup.password` is acorn-parseable (with `@` plugin). Pre-processor must recognize the validator-attribute form during decl extraction.
- Edge: refinement-type predicates `<email>: string(pattern(/^[^@]+@[^@]+$/)) req = <input/>` â€” type annotation `string(pattern(...))` is a scrml-level form; the regex literal `/^[^@]+@[^@]+$/` is JS-level (acorn handles regex literals fine).
- Edge: `reset(@cell)` keyword call â€” acorn would see `reset` as identifier-call. Two paths: (a) handle in pre-processor (extract `reset(@cell)` as a special kind), or (b) let acorn parse as a regular call and post-process based on the keyword token. (b) is simpler and aligns with how `lift` is handled today.

**Conclusion: acorn stays.** Cost is in the pre-processor (Â§A.3), not in acorn replacement.

**Cost:** ~0-3h direct (no acorn replacement). Possibly small custom-plugin updates if `@` handling needs new edge cases.
**Dependencies:** Â§A.3.
**Risk:** low. acorn architecture is sound.

## Â§I Recommendation: piecemeal migration, not greenfield

**Why piecemeal:**
1. Parser + codegen + runtime are well-structured with discrete passes per PIPELINE.md. Per-pass migration is feasible.
2. ~60% of the parser is v0.next-compatible already (JS expression body, block-splitter mechanics for non-decl-form blocks, markup tag parsing for static HTML, type-decl synthesis, codegen for non-engine non-validator surfaces).
3. Discarding 6+ months of working code (~24,739 LOC compiler + 8,769 tests + 32 examples + 277 samples + ~12,048 LOC self-host) for a rewrite ignores the partial-fit and accumulates rework cost (the rewrite itself may take 200-400h before it equals current functionality).
4. v0.1.0 syntax (`@NAME = init` inside `${}`) can coexist with v0.next syntax in the parser â€” strict-non-deprecation is feasible for the migration window.

**Why NOT greenfield rewrite:**
- Estimated rewrite cost: ~400-800h (build new compiler from scratch to feature-parity with current).
- Loses test corpus, sample corpus, examples, self-host parity, community-published v0.1.0 code.
- Migration semantics for v0.1.0 â†’ v0.2.0 must still be implemented; rewrite doesn't avoid the cost.

**Decision:** PIECEMEAL via Phase A â†’ B â†’ C structure above.
