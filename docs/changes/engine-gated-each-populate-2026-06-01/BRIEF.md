# THE BUG (empirically confirmed by PA at HEAD 4e1f9492)

An `<each>` whose mount lives inside a NON-`initial=` engine state-child never populates. The each-mount `<div data-scrml-each-mount="each_N">` is emitted INSIDE the arm's render output (e.g. `_scrml_engine_phase_render_Browsing()`), so at module-init it is absent from the DOM (the engine renders only the `initial=` arm). Two coupled failure modes:

**Mode 1 — dep never tracked.** The emitted each render fn (`emitEachBodyRenderForFile` in `compiler/src/codegen/emit-each.ts`, ~line 828-856) is:
```js
function _scrml_each_render_N() {
  const _mount = document.querySelector('[data-scrml-each-mount="each_N"]');
  if (!_mount) return;                       // <-- returns BEFORE reading the cell
  const _items = _scrml_reactive_get("todos");
  ...
}
```
The dispatcher (`emit-each.ts` ~line 919-920) emits `_scrml_each_render_N(); _scrml_effect_static(_scrml_each_render_N);`. **`_scrml_effect_static` collects deps ONLY on its FIRST run** (`hasRun` flag in `compiler/src/runtime-template.js` — `if (hasRun) { fn(); return; }`). At module-init the engine is in `initial=` (mount absent) → the first run hits `if (!_mount) return;` BEFORE `_scrml_reactive_get("todos")` → ZERO deps tracked → the effect is permanently subscribed to nothing and never re-fires.

**Mode 2 — no render on arm-entry.** The engine dispatcher (`__scrml_engine_phase_dispatch` in emitted client.js, generated by `compiler/src/codegen/emit-engine.ts`) on entering `.Browsing` does `_mount.innerHTML = _scrml_engine_phase_render_Browsing(); _scrml_engine_phase_wire_Browsing(_mount);`. The each-mount div now exists in the DOM, but **nothing calls `_scrml_each_render_N()`**, so the list renders empty.

This is one instance of the broader v1 limitation documented in PRIMER §7 (non-initial engine arm bodies are static HTML snapshots; reactive content inside them isn't re-wired across variant changes). NO SPEC change is needed — SPEC §17.7 + §51.0 intend the each to populate; this is a codegen gap.

# THE FIX — both parts required

**Part A — `compiler/src/codegen/emit-each.ts`: read the source before the mount guard.** Reorder the emitted render fn so the iteration-source resolution (`const _items = ...` for `in=`, and the count resolution for `of=`) happens BEFORE `const _mount = ...querySelector...; if (!_mount) return;`. The `_scrml_reactive_get(...)` read MUST execute on the first `_scrml_effect_static` run regardless of mount presence, so the dep is always established. After computing `_items`, query `_mount`; if absent, `return` (can't render, but the dep is now tracked). The `<empty>`/`!_items` guards stay after `_items` is resolved (they already use `_items`). This changes the emitted shape of EVERY each render fn (reordering two statements) — equivalence in the normal (mount-present) case is preserved.

**Part B — engine dispatcher re-invokes each render fns on arm-entry.** After the engine dispatcher writes an arm's `innerHTML` (and wires it), the each render fn(s) for each-mounts in that arm must run. **Recommended design (decoupled registry + generic remount helper) — but SURVEY the engine codegen first and choose the cleanest integration; the constraints below are what matters, not the mechanism:**
- emit-each.ts registers each renderer in a global map: `_scrml_each_renderers["each_N"] = _scrml_each_render_N;` (alongside the existing dispatcher emission).
- A small runtime helper (add to `compiler/src/runtime-template.js` or the appropriate runtime-chunk) — e.g. `_scrml_remount_each(root)` — does `root.querySelectorAll('[data-scrml-each-mount]').forEach(el => { const id = el.getAttribute('data-scrml-each-mount'); const fn = _scrml_each_renderers[id]; if (fn) fn(); });`.
- The engine dispatcher (emit-engine.ts) calls `_scrml_remount_each(_mount)` after `_mount.innerHTML = render_X(); wire_X(_mount);` for every arm.
- This decoupled approach generalizes (handles nested eaches via querySelectorAll; reusable by match-block dispatch later) and avoids threading each-ids through the engine codegen. If the engine codegen structure makes a more direct hook cleaner (e.g. emitting the each render call inside `wire_X`), that is acceptable PROVIDED it satisfies all four constraints:
  1. **Data-present-on-entry:** entering an arm with the source cell already populated renders the list.
  2. **Ongoing reactivity after entry:** mutating the source cell while the arm is visible re-renders (Part A makes this work via the now-established effect dep — verify it actually does).
  3. **Nested eaches:** an each nested several levels inside the arm body still renders.
  4. **Idempotent / dispose-safe:** re-entering an arm (Loading→Browsing→Loading→Browsing) re-renders correctly; no duplicate subscriptions or leaks. Mind the engine's `__scrml_engine_phase_dispose` lifecycle — if a renderer-registry entry needs cleanup, wire it; if the each effect's dep is module-level (it is — `_scrml_effect_static` subscribed once at module-init), confirm re-entry doesn't double-fire or leak.

**Scope boundary:** fix the ENGINE dispatcher path (the req2 case). The same class likely affects match-block dispatch (`emit-variant-guard.ts`) and other dynamic-HTML insertion sites — note these as follow-ups in progress.md, but do NOT expand scope to them unless your registry helper makes the engine fix trivially cover them (in which case, call the helper at those sites too and note it). Do NOT touch the `:`-shorthand-in-engine-arm parser bug.
