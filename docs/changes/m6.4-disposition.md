# M6.4 — ast-builder P2-Form1 splitBlocks disposition

## Headline

**Disposition: NEITHER (a) deletion NOR (b) migration — STOP, escalate for unit-split.**

The native parser does NOT bypass `_splitBlocksForP2Form1` cleanly: end-to-end CLI compile with
`--parser=scrml-native` on a P2-Form1 exemplar emits `E-COMPONENT-035` ("Component survived CE
but was not resolved"). The native-side mirror in `parse-markup.js` is incomplete for the
P2-Form1 pair-detection branch — it does not synthesize the `export-decl` that MOD/CE need to
register the exported component name.

## Call-site survey (the easy part)

`compiler/src/ast-builder.js:893` — `_splitBlocksForP2Form1` is an aliased re-export of
`splitBlocks` (`import { splitBlocks as _splitBlocksForP2Form1 } from "./block-splitter.js"`
at line 49). It is invoked from EXACTLY ONE site (verified by grep), inside
`liftBareDeclarations` (line 740) which is called from inside `buildAST` (line 11999).

**Upstream gate:** `--parser=scrml-native` (api.js:843-850) replaces `_buildAST` entirely with
`nativeParseFile`. The live `buildAST` is never called; therefore `liftBareDeclarations` is
never called; therefore `_splitBlocksForP2Form1` is never reached. This part of the cutover
plan's claim is **architecturally correct**.

**Native-side mirror:** `compiler/native-parser/parse-markup.js:1897-2318` contains the
explicit native mirror of `liftBareDeclarations` (multiple in-file comments confirm
"Mirrors liftBareDeclarations 1:1"). The pair-detection branches are visible at lines 2055,
2237, 2318 — but the P2-Form1 path's `export <ComponentName ...>...</>` synthesis is
incomplete: the native pipeline emits a `markup` block instead of synthesizing the
`logic`-block-with-`export-decl` structure that downstream stages consume.

## P2-Form1 exemplar + canary evidence

Two exemplars used (both shapes asserted as canonical by
`compiler/tests/unit/p2-export-component-form1.test.js` §A):

### Exemplar 1 — cross-file (mirrors integration test §X1)

```
// components.scrml
export <X1Badge>
  <span class="x1-badge">badge</>
</>

// app.scrml
<program>
${ import { X1Badge } from './components.scrml' }
<div>
  <X1Badge/>
</div>
</program>
```

| Pipeline | Errors | Warnings | `x1-badge` in artifact |
|---|---|---|---|
| LIVE   (default) | 0 | 2 | YES |
| NATIVE (`--parser=scrml-native`) | **2** | 1 | NO |

Native errors (verbatim, both fired):
- `E-COMPONENT-035: Component \`X1Badge\` survived component expansion (CE) but was not resolved. This is a post-CE invariant violation: every markup node resolved to user-component (or unresolved with an [explicit error])`

### Exemplar 2 — single-file inside `<program>`

```
<program>
  export <Greeting props={ name: string }>
    <p class="hello">Hello, ${name}!</>
  </>
</program>
```

| Pipeline | Errors | Warnings | `Greeting` / `hello` in artifact |
|---|---|---|---|
| LIVE   | 0 | 1 | LIVE wraps the component body in a way the live HTML asserter doesn't see |
| NATIVE | **1** | 1 | NATIVE emits the markup but fails to resolve |

Native error: `E-COMPONENT-035: Component \`Greeting\` survived component expansion (CE) but was not resolved.`

### TAB-only canary (via `classifyDivergence`)

The conformance canary classifies these as `LIVE-DEGENERATE` (the heuristic credits NATIVE as
correct). **The canary's heuristic is over-broad here:** the SPEC §21.2 + the unit-test
contract assert the LIVE shape (synthetic logic block with `export-decl`, `ast.exports.length === 1`,
`exportedName === "Greeting"`). NATIVE produces `exports: []` (no export entry, no synthesis).

This is the canary class-design boundary the corpus-sweep ledger has been tracking; for M6
purposes the empirical truth is the end-to-end CLI compile result above.

## Why neither disposition is clean

### (a) Deletion candidate — REJECTED

The bypass IS architecturally clean (native mode never calls `_splitBlocksForP2Form1`), but
the bypass is INCOMPLETE — the native side does not provide an equivalent. Deleting the live
call-site (or the entire helper) would not fix the native-mode regression; the regression
exists today even with the live call-site fully in place, because native mode does not call
into the live path.

### (b) Pattern-match migration to `parseMarkup` — NOT VIABLE

The M6.1 pattern (replace `splitBlocks` synthesis with `parseMarkup`) does not apply here
because the call-site is NOT a "synthesis re-invocation that bypasses native routing." It is
a synthesis step that lives INSIDE the legacy `liftBareDeclarations` post-pass, which is
itself bypassed in native mode. Migrating it changes the live path; the native path is the
broken one.

The actual fix needs to live in `compiler/native-parser/parse-markup.js` — extending the
pair-detection mirror (lines 2055/2237/2318) to handle the P2-Form1 `export <PascalCase>`
case by synthesizing the equivalent `export-decl`-bearing native Block. That is a native-
parser-side change, NOT a `splitBlocks`-migration change.
